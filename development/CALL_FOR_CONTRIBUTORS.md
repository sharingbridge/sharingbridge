# Join Us in Building SharingBridge — Community meal coordination

## 🌟 Project Overview

**SharingBridge** is an open-source, AI-driven initiative to transform meal support by creating a transparent, dignified platform that connects payees with people who need meals through verified food delivery services. We ensure cash handouts reach their intended purpose - providing food and essentials rather than cash that could be misused.

**Our Mission:** Make affordable meals easier to arrange and pay for—while maintaining dignity for everyone involved.

**Before you contribute:** Read [SharingBridge_Business_Requirement.md](../requirements/SharingBridge_Business_Requirement.md), section **“Operating Constraints & Assumptions.”** That section is the single source of truth for product-level assumptions.

## 🤖 Development Approach: AI-Assisted Collaboration

**SharingBridge is being built through AI-assisted coding sessions.** Most code, docs, and infrastructure artifacts are generated by humans pairing with AI assistants. This means:

- ✨ **You don't need to be a coding expert** to contribute meaningfully
- 💬 **Contribution happens through GitHub Discussions** - describing what you need in natural language
- 🤝 **Diverse expertise matters more than technical skills** - we need humanitarian perspectives, legal guidance, and community insights
- 📋 **Live coordination** - the next set of recommended tasks for the AI-assisted track lives in [development/AGENT_HANDOFF.md](AGENT_HANDOFF.md)

**This changes everything.** Whether you're a social worker, lawyer, community organizer, or concerned citizen - your domain knowledge is just as valuable as coding skills.

## 🤝 Why This Matters

Every day, people meet those needing a meal and face a difficult choice - help with cash that may be misused, or walk away. SharingBridge solves this by:

- ✅ Turning payment into **meals and essentials**, not ambiguous cash
- ✅ Providing delivery confirmation with photo proof
- ✅ Maintaining dignity for payees, initiators, and people receiving meals
- ✅ Leveraging existing delivery infrastructure
- ✅ Creating community solidarity through pledge pools and crowdfunding

**Impact Potential:** Millions of meal-support interactions yearly could become transparent, accountable acts of kindness.

## 📁 Repository Structure

**Independent Evolution. Coordinated Progress.**

SharingBridge uses a **multi-repo architecture** where each stream evolves independently. Development is largely AI-assisted; the live coordination doc is [development/AGENT_HANDOFF.md](AGENT_HANDOFF.md).

### Master Repository (This Repo)
- **Purpose:** Overall coordination, documentation, requirements
- **Contains:** Business requirements, architecture docs, manual testing guide, MVP bootstrap checklist, agent handoff
- **Track:** Links to all child repositories and their status

### Child Repositories (Independent Streams)

Each repository contains:
- Source code and artifacts
- Issues and discussions specific to that component
- Independent CI/CD pipelines

**Frontend Repositories:**
- `sharingbridge-mobile-app` - Flutter mobile application
- `sharingbridge-web-app` - React/Next.js web application

**Backend Repositories:**
- `sharingbridge-api-gateway` - API gateway and routing
- `sharingbridge-order-service` - Order management microservice
- `sharingbridge-user-service` - User authentication and profile service
- `sharingbridge-integration-service` - Vendor integration (Swiggy, Zomato, etc.)
- `sharingbridge-notification-service` - Push notifications and alerts

**AI/ML Repositories:**
- `sharingbridge-photo-service` - Face detection and beneficiary assistance history matching
- `sharingbridge-ai-orchestration` - LLM instruction-pack and vendor suggestions
- `sharingbridge-location-safety` - **Archived** (MVP uses mobile handover guidance only)

**Infrastructure Repositories:**
- `sharingbridge-infra` - Infrastructure as Code (Terraform/CloudFormation)
- `sharingbridge-deployment` - CI/CD pipelines and deployment scripts

**Note:** Documentation is maintained within each service repository rather than in a separate docs repo. User guides go in mobile/web repos, API docs in service repos, deployment guides in infra repos.
